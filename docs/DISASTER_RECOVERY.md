# خطة التعافى من الكوارث

> آخر تحديث: ٢٠ يوليو ٢٠٢٦ — الإصدار 3.74.770

---

## الحقيقة الحالية

| المصدر | الحالة |
|---|---|
| نسخ Supabase التلقائية | **غير مفعّلة** — الخطة المجانية لا تشملها |
| `supabase/schema/*.sql` | **لا تُعيد البناء** — أُثبت بالاختبار |
| نسخة التطبيق (الإعدادات ← النسخ) | **بيانات شركة واحدة، بلا بنية** — أداة لمشكلة أخرى |
| `backups/` عبر `pg_dump` | **الخطة الفعلية** |

---

## نسخة التطبيق ليست بديلاً — وليست منافساً

سؤالٌ يستحق إجابة صريحة، لأن التطبيق فيه ميزة نسخ تعمل جيداً:

| | نسخة التطبيق | `pg_dump` |
|---|---|---|
| الجداول | **١٥٥** من ٢٤٩ | **٢٤٩** |
| البيانات | نعم | نعم |
| **الدوال (١٢٠٤)** | **صفر** | كاملة |
| **المشغّلات (٥٠١)** | **صفر** | كاملة |
| **سياسات الحماية (٧٩٧)** | **صفر** | كاملة |
| النطاق | **شركة واحدة** | كل الشركات |
| الصيغة | JSON | SQL |
| مستثنى | `audit_logs`، `company_invitations`، `restore_queue` | لا شىء |

**نسخة التطبيق تنسخ البيانات، لا البنية.** وهى تستعيد **إلى قاعدة موجودة**.

فإن ضاعت قاعدة البيانات نفسها، **لا يوجد ما تُستعاد إليه**: لا جداول تستقبل
الصفوف، ولا مشغّلات تحسب التكلفة، ولا سياسات تفصل بين شركاتك.

| السيناريو | ما ينقذك |
|---|---|
| «حذفتُ عملائى بالخطأ» | ✅ **نسخة التطبيق** |
| «شركة فقدت بياناتها» | ✅ **نسخة التطبيق** |
| «قاعدة البيانات ضاعت أو تلفت» | ❌ نسخة التطبيق عاجزة ← **`pg_dump`** |

**الأولى تحمى بياناتك. الثانى يحمى نظامك.** ولا يُغنى أحدهما عن الآخر.

**وانتبه:** `audit_logs` مستثنى من نسخة التطبيق عمداً — **سجل التدقيق لا يُنسخ
فيها إطلاقاً**، وهو أول ما يطلبه مدقق.

قبل ٢٠ يوليو ٢٠٢٦ لم تكن هناك نسخة احتياطية عاملة **إطلاقاً**. لا منقوصة — معدومة.
٢٧٠ ميجابايت من بيانات محاسبية حيّة، ٢٤٩ جدولاً، نحو ٢٤٠ ألف صف، ولا شىء يُعيدها.

---

## لماذا لا تصلح لقطة المستودع

استُعيدت فى مشروع نظيف يوم ٢٠ يوليو. النتيجة:

```
١٠٩١٦ عبارة    ٤٨٧١ نجحت    ٦٠٤٥ فشلت

الجداول    ٢٤٩ ← ٢٤٣        السياسات  ٧٩٧ ← ٤٢٩
الدوال    ١٢٠٤ ← ١٠٥٣       المشغّلات  ٥٠١ ← ٥
```

**خمسة مشغّلات من ٥٠١.** المشغّلات موضع ترحيل التكلفة، وفرض توازن القيود،
واستهلاك دفعات FIFO. قاعدة بلا مشغّلات **تقبل البيانات وتتوقف عن المحاسبة**.

**وأخطر ما فى النتيجة أن ٤٨٧١ عبارة نجحت** — فالنظام المُستعاد كان **سيبدو عاملاً**.

الأسباب بنيوية: لا ترتيب اعتماديات، ولا امتدادات، ولا أنواع مخصصة، ولا متتاليات.
كلها أمورٌ يحلّها `pg_dump` **بحكم تصميمه**.

**اللقطة تبقى نافعة كسطح مقارنة** — وقد أثبتت ذلك يوم ٢٠ يوليو حين كشفت نسخة
كانت ما زالت تمنح ثلاث دوال خطرة لغير المسجَّل بعد حذفها.

## التهيئة — مرة واحدة

```bash
npx supabase login
npx supabase link --project-ref hfvsbsizokxontflgdyn
npm install pg --save-dev
```

خطوة الربط تطلب **كلمة مرور قاعدة بيانات الإنتاج**.

> **مهم:** هذه ليست كلمة المرور التى يستخدمها تطبيقك — التطبيق يتصل بمفاتيح API.
> إن لم تكن مسجَّلة عندك، أعد تعيينها من اللوحة (Connect ← Direct ← Reset password).
> **إعادة التعيين لا تُعطّل التطبيق العامل.**

## النسخ

```bash
node scripts/backup-production.js
```

يُنتج `backups/<التاريخ>/` وفيه:

| الملف | المحتوى |
|---|---|
| `roles.sql` | تعريفات الأدوار |
| `schema.sql` | البنية: جداول، أنواع، دوال، سياسات حماية |
| `data.sql` | الصفوف |

**السكربت يرفض:**
- إن لم يكن مربوطاً بالإنتاج
- إن فشل أى جزء (ويقول إن النسخة **ناقصة**)
- إن كان الناتج أصغر من ٥ ميجابايت مقابل قاعدة ٢٧٠ ميجابايت — **ملفٌ صغير ينجح بلا خطأ أخطر من فشل صريح**

## التحقق — وهذا هو الجزء الذى يُهمل عادةً

```bash
node scripts/verify-backup.js backups/<التاريخ>
```

يستعيد النسخة فى **قاعدة الاختبار** ثم يقارن الأعداد بالإنتاج.

> **النسخة الاحتياطية ليست نسخة حتى تُستعاد.**
>
> هذه خلاصة يوم ٢٠ يوليو. لقطة المستودع اعتُبرت خطة تعافٍ لشهور، وأول مرة
> جُرّبت فعلياً فشلت. **عدّ العبارات الناجحة يخبرك أن الملف قُرئ. مقارنة النتيجة
> بالإنتاج تخبرك هل تستطيع العمل عليها غداً.**

## الوتيرة المقترحة

| متى | ماذا |
|---|---|
| **أسبوعياً** | نسخة كاملة |
| **قبل أى إصدار كبير** | نسخة + تحقق |
| **شهرياً** | تحقق كامل من آخر نسخة |
| **عند بلوغ الإيراد** | ترقية إلى Pro — نسخ يومية تلقائية تُديرها المنصة |

**احفظ النسخ خارج جهازك** — قرص خارجى أو تخزين سحابى. نسخة على نفس الجهاز الذى
قد يتعطّل ليست نسخة.

**ومجلد `backups/` مستثنى من git عمداً** — يحوى كل صف من بيانات عملائك. رفعه
يعنى نشرها.

## ⚠️ المراجع الدائرية — لا تتجاهل هذه الخطوة

`pg_dump` حذّر ستّ عشرة مرة أثناء النسخ:

```
warning: there are circular foreign-key constraints among these tables:
   companies, chart_of_accounts
   invoices, sales_orders
   purchase_orders, bills, goods_receipts
hint: You might not be able to restore the dump without --disable-triggers
```

**المعنى:** جداولك مترابطة دائرياً — الفواتير تشير لأوامر البيع، وأوامر البيع
تشير للفواتير. فلا يوجد **ترتيب** لإدخال الصفوف يُرضى كل المراجع أثناء التحميل:
أيًّا كان الجدول الأول، سيشير إلى صفوف لم تُدخل بعد.

**النتيجة لو تُجوهل:** `schema.sql` يُستعاد بلا خطأ، و**`data.sql` يفشل**. نسخة
«تعمل» فى نصفها — وهو النمط الذى تدور حوله هذه الوثيقة كلها.

**الحل عند الاستعادة اليدوية:**

```sql
SET session_replication_role = 'replica';   -- قبل data.sql
```

يُوقف فحص المراجع والمشغّلات **لهذه الجلسة فقط** أثناء التحميل. ثم:

```sql
SET session_replication_role = 'origin';    -- بعده
```

`verify-backup.js` يفعل هذا تلقائياً، **ويفحص بعده أن كل المراجع صارت مُتحقَّقة** —
فالفحص مُؤجَّل لا مُلغى، والبيانات ما زالت مُلزَمة بأن تكون سليمة.

## عند وقوع الكارثة

1. أنشئ مشروع Supabase جديداً
2. `psql <رابط الجديد> -f backups/<التاريخ>/roles.sql`
3. ثم `schema.sql`
4. ثم `data.sql` **مسبوقاً بـ`SET session_replication_role = 'replica';`** — انظر القسم أعلاه
4. حدّث `.env.local` و متغيرات Vercel بالمفاتيح الجديدة
5. شغّل `node scripts/check-schema-snapshot-fresh.js` للتأكد من التطابق

**جرّب هذه الخطوات مرة واحدة على قاعدة الاختبار قبل أن تحتاجها.** الفرق بين خطة
مكتوبة وخطة مُجرَّبة هو الفرق الذى كشفه يوم ٢٠ يوليو.
