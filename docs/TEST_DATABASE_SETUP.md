# إعداد قاعدة بيانات الاختبارات

> **v3.74.745** — هذا المستند يُغلق آخر بند فى خارطة الطريق التى وضعناها فى ٢٠ يوليو ٢٠٢٦.

## لماذا نحتاج مشروعاً منفصلاً

اختبارات `tests/integration` و`tests/e2e` **تُنشئ مستخدمين وشركات وفواتير ومدفوعات وقيوداً محاسبية، ثم تحذفها**.

قبل **v3.74.740** كانت تقرأ `NEXT_PUBLIC_SUPABASE_URL` و`SUPABASE_SERVICE_ROLE_KEY` — **نفس متغيرَى التطبيق**. فلحظة أن يُضاف هذان السرّان إلى GitHub «لتشغيل الاختبارات»، يبدأ كل دفع بكتابة شركات وهمية داخل قاعدة المحاسبة الحية.

**لم يحدث ذلك** — تحققنا: صفر مستخدم بنمط `test-%@test.com` وصفر شركة بنمط `%test%` فى الإنتاج. **كان لغماً لم ينفجر.**

الآن الاختبارات **تُسمّى قاعدتها بنفسها** ولا ترتد إلى متغيرات التطبيق، **وترفض معرّف مشروع الإنتاج صراحةً**.

## ماذا يفتح هذا البند

| البند | الحال اليوم |
|---|---|
| ٥٨ حالة اختبار | معطَّلة — لا تعمل ولا تحرس شيئاً |
| النسخة المرجعية للقاعدة | **محفوظة، غير مُثبَتة** |

والثانى هو الأهم: `supabase/schema/schema.sql` و`functions.sql` يحفظان الجداول والسياسات والمُشغِّلات والصلاحيات — **لكن لم يُثبَت قط أنهما يُعيدان بناء القاعدة**. والفرق بين «نرى ما فى الإنتاج» و«نستطيع استعادته» هو الفرق بين **توثيق** و**تعافٍ من كارثة**.

**استعادة النسخة فى مشروع فارغ تُثبت الأمرين معاً.**

## الخطوات

### ١. أنشئ مشروع Supabase جديداً

من لوحة تحكم Supabase. سمِّه شيئاً لا يُخلَط بالإنتاج، مثل `erb-vitaslims-test`.

> **تنبيه تكلفة:** المشاريع المجانية تُوقَف تلقائياً بعد فترة خمول، وقد تحتاج خطة مدفوعة إن أردته دائم التشغيل. هذا قرارك.

### ٢. استعد المخطط ثم الدوال — **بهذا الترتيب**

من محرر SQL فى المشروع الجديد:

1. الصق محتوى `supabase/schema/schema.sql` ونفّذه
2. ثم الصق محتوى `supabase/schema/functions.sql` ونفّذه

**الترتيب مهم:** الدوال تشير إلى جداول، لا العكس.

> **وهذه هى لحظة الإثبات.** إن نجحت الاستعادة، فالمستودع يُعيد بناء قاعدة البيانات فعلاً. وإن فشلت، **فقد اكتشفنا ذلك على مشروع فارغ لا على كارثة حقيقية** — وهو بالضبط الغرض.

### ٣. أضف المتغيرين محلياً

فى `.env.local`:

```
TEST_SUPABASE_URL=https://<ref-الجديد>.supabase.co
TEST_SUPABASE_SERVICE_ROLE_KEY=<service-role-key-الجديد>
```

> ⚠️ `.env.local` مُستثنى من Git. **لا تضع هذه القيم فى أى ملف يُلتزَم.**

### ٤. تحقق قبل أن تُشغّل شيئاً

```bash
npm run test:db:verify
```

يفحص أربعة أشياء **قبل** أن يلمس أى اختبار القاعدة:

- أن الهدف **ليس الإنتاج**
- أنه قابل للوصول والمفتاح صحيح
- أن **المخطط موجود** — وإلا لظهرت كل الاختبارات فاشلة وبدت كأنها عيوب فى الكود
- أن الاستعادة **مكتملة** لا جزئية

### ٥. شغّل الاختبارات

```bash
npm run test:integration
npm run test:e2e
```

### ٦. فعّلها فى CI

أضف فى GitHub → Settings → Secrets:

- `TEST_SUPABASE_URL`
- `TEST_SUPABASE_SERVICE_ROLE_KEY`

عندها **يتوقف CI عن قول «٥٨ حالة لا تعمل»**، وتصير الطبقتان بوابة نشر حقيقية — تفشل فيُوقَف النشر.

## ما لا يجب فعله

- **لا تضع مفاتيح الإنتاج** فى `TEST_SUPABASE_*`. `tests/helpers/test-setup.ts` سيرفض، لكن لا تعتمد على ذلك وحده.
- **لا تُعِد `continue-on-error`** إلى خطوات الاختبار. تشغيل شىء تتجاهل نتيجته يشترى شعوراً بالتغطية ولا يدفع ثمنه.
- **لا تستخدم قاعدة الاختبار لبيانات حقيقية.** الاختبارات تحذف ما تُنشئه، وقد تحذف غيره.
